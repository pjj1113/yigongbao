# 生产管理设计文档

## 1. 业务概述

生产管理模块负责管理从设计订单分配而来的生产任务。设计审核通过后，系统根据设计订单中的**数据包**自动生成，每个数据包对应一条生产记录。

**生产批号规则**：格式为 编号 + '-' + 数据包序号`，例如同一订单有 2 个数据包则生成：
- `20260513000001-1`
- `20260513000001-2`

其中 `20260513000001` 为该订单对应的系统编号（每个订单取一次），`-1`、`-2` 为数据包序号。

**触发时机**：设计审核通过（`order_main.phase=20, status=2030` → `phase=30, status=3010`）时，后端自动为每个 `design_package` 创建一条 `production_order` 记录。

## 2. 数据模型

### 2.1 production_order（生产订单表）

```sql
DROP TABLE IF EXISTS production_order;
CREATE TABLE production_order (
    id                  BIGINT          NOT NULL AUTO_INCREMENT COMMENT '主键ID',

    -- ==================== 生产批号 ====================
    production_no       VARCHAR(80)     NOT NULL COMMENT '生产批号',

    -- ==================== 来源关联 ====================
    order_id            BIGINT          NOT NULL COMMENT '订单ID（order_main.id）',
    order_code          VARCHAR(50)     NOT NULL COMMENT '订单编号（冗余）',
    package_id          BIGINT          NOT NULL COMMENT '数据包ID（design_package.id）',
    package_code        VARCHAR(50)     NOT NULL COMMENT '数据包编号（冗余）',
    package_seq         INT             NOT NULL COMMENT '数据包序号（冗余）',

    -- ==================== 订单冗余信息（避免跨表查询）====================
    org_id              BIGINT          COMMENT '提单机构ID',
    org_name            VARCHAR(200)    COMMENT '提单机构名称',
    hospital_id         BIGINT          COMMENT '医院ID',
    hospital_name       VARCHAR(200)    COMMENT '医院名称',
    hospital_dept_name  VARCHAR(100)    COMMENT '科室名称',
    area_name           VARCHAR(64)     COMMENT '地区名称',
    doctor_name         VARCHAR(100)    COMMENT '医生姓名',
    patient_name        VARCHAR(100)    COMMENT '患者姓名',
    is_urgent           TINYINT         DEFAULT 0 COMMENT '是否加急：0-否，1-是',
    is_postal           TINYINT         DEFAULT 0 COMMENT '是否邮寄：0-否，1-是',
    expected_delivery_date DATETIME     COMMENT '期望交付时间',
    order_type          TINYINT         COMMENT '订单类型：1-医疗器械，2-非医疗器械',
    business_type       VARCHAR(20)     COMMENT '业务类型（字典 dict_code：11.x）',

    -- ==================== 数据包冗余信息 ====================
    product_mark        VARCHAR(100)    COMMENT '产品标识（来自 design_package.product_mark）',
    pack_quantity       INT             COMMENT '包装数量（来自 design_package.pack_quantity）',

    -- ==================== 生产状态 ====================
    -- 3010=设计完成, 3020=待打印, 3030=打印中, 3040=打印完成, 3050=后处理, 3060=打包, 3070=入库
    status              INT             NOT NULL DEFAULT 3010 COMMENT '生产状态',

    -- ==================== 生产分配 ====================
    producer_id         BIGINT          COMMENT '生产负责人ID',
    producer_name       VARCHAR(100)    COMMENT '生产负责人姓名',
    center_id           BIGINT          COMMENT '生产中心ID（production_center.id）',
    center_name         VARCHAR(100)    COMMENT '生产中心名称',

    -- ==================== 时间节点 ====================
    plan_start_time     DATETIME        COMMENT '计划开始时间',
    actual_start_time   DATETIME        COMMENT '实际开始时间',
    actual_finish_time  DATETIME        COMMENT '实际完成时间',

    -- ==================== 备注 ====================
    remark              VARCHAR(500)    COMMENT '备注',


    -- ==================== 公共字段 ====================
    create_time         DATETIME        DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time         DATETIME        DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    create_by           BIGINT          COMMENT '创建人ID',
    update_by           BIGINT          COMMENT '更新人ID',
    is_deleted          TINYINT         DEFAULT 0 COMMENT '是否删除（0=否，1=是）',

    PRIMARY KEY (id),
    KEY idx_production_order_order_id (order_id),
    KEY idx_production_order_package_id (package_id),
    KEY idx_production_order_status (status),
    KEY idx_production_order_create_time (create_time)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='生产订单表';

CREATE UNIQUE INDEX uk_production_no ON production_order ((CASE WHEN is_deleted = 0 THEN production_no ELSE NULL END));
```

记录每条生产订单的各工序执行情况。

```sql
DROP TABLE IF EXISTS production_process_record;

    production_order_id BIGINT          NOT NULL COMMENT '生产订单ID',
    production_no       VARCHAR(80)     NOT NULL COMMENT '生产批号（冗余）',

    -- ==================== 工序信息 ====================
    process_type        VARCHAR(50)     NOT NULL COMMENT '工序类型（字典：process_type）',
    process_name        VARCHAR(100)    NOT NULL COMMENT '工序名称（冗余）',
    process_seq         INT             NOT NULL COMMENT '工序顺序',

    -- ==================== 设备分配 ====================
    device_id           BIGINT          COMMENT '分配设备ID（production_device.id）',
    device_no           VARCHAR(50)     COMMENT '设备编号（冗余）',
    device_name         VARCHAR(100)    COMMENT '设备名称（冗余）',

    -- ==================== 执行信息 ====================
    -- 0=待执行, 1=执行中, 2=已完成, 4=质检
    status              TINYINT         DEFAULT 0 COMMENT '工序状态',
    operator_id         BIGINT          COMMENT '操作人ID',
    operator_name       VARCHAR(100)    COMMENT '操作人姓名',
    start_time          DATETIME        COMMENT '开始时间',
    finish_time         DATETIME        COMMENT '完成时间',
    remark              VARCHAR(500)    COMMENT '备注',

    -- ==================== 公共字段 ====================
    create_time         DATETIME        DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time         DATETIME        DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    create_by           BIGINT          COMMENT '创建人ID',
    update_by           BIGINT          COMMENT '更新人ID',
    is_deleted          TINYINT         DEFAULT 0 COMMENT '是否删除（0=否，1=是）',

    PRIMARY KEY (id),
    KEY idx_ppr_production_order_id (production_order_id),
    KEY idx_ppr_status (status)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci COMMENT='生产工序记录表';
```

## 3. 状态流转

### 3.1 生产订单状态

| 状态码 | 名称 | 说明 |
|--------|------|------|
| 3010 | 设计完成 | 设计审核通过后自动创建，等待打印 |
| 3020 | 待打印 | 已分配生产中心/设备，等待开始打印 |
| 3030 | 打印中 | 打印工序执行中 |
| 3040 | 打印完成 | 打印工序完成，进入后处理 |
| 3050 | 后处理 | 清洗/固化/干燥等后处理工序执行中 |
| 3060 | 打包 | 后处理完成，执行包装工序 |
| 3070 | 入库 | 所有工序完成 |

### 3.2 工序状态

| 状态码 | 名称 |
|--------|------|
| 0 | 待执行 |
| 1 | 执行中 |
| 2 | 已完成 |
| 3 | 质检 |

### 3.3 生产工序顺序（标准流程）

| 序号 | 工序类型 | 工序名称 | 说明 |
|------|----------|----------|------|
| 1 | PRINT | 3D打印 | 由打印机自动执行，完成由 WebSocket 推送，无需人工操作 |
| 2 | WASH | 酒精清洗/打磨（含干燥） | 酒精清洗、打磨及干燥合并为一道工序 |
| 3 | CURING | UV固化 | |
| 4 | ULTRASONIC | 超声清洗/干燥 | |
| 5 | PACK | 包装 | |

> **注：** 原 DRY（干燥）工序已合并入 WASH，干燥参数（温度、时长）作为 WASH 工序的可选参数填写。

### 3.4 非医疗器械产品流程说明

`order_type = 2`（非医疗器械）的生产订单，打印完成后**跳过所有后处理工序**（WASH / CURING / ULTRASONIC）及终检，直接进入包装（PACK）。

```
3030 打印中 →[PRINT_DONE]→ 3040 打印完成 →[直接]→ PACK → 3070 入库
```

后端在 PRINT 工序质检合格时，检查 `order_type`：
- `order_type = 1`：正常推进到 WASH（order.status → 3050）
- `order_type = 2`：跳过后处理，order.status → 3060（后处理完成），直接进入 PACK 工序

### 4.3 下载数据包（触发状态变更）

**POST** `/production/order/{id}/download-package`

**说明：** 点击下载数据包时调用，将生产订单状态由 `3010（设计完成）` 变更为 `3020（待打印）`。仅当当前状态为 `3010` 时执行变更，其他状态直接返回成功（幂等）。

**路径参数：**

| 参数 | 类型 | 说明 |
|------|------|------|
| id | Long | 生产订单ID |

**响应：**
```json
{
  "code": 200,
  "msg": "success",
  "data": null
}
```

**状态变更：** `production_order.status: 3010 → 3020`

**POST** `/production/order/{id}/assign-device`

**说明：** 用户选中打印机后提交，将该打印机绑定到生产订单，记录到 `production_process_record` 的 PRINT 工序中。

**请求体：**
```json
{
  "deviceId": 1
}
```

**响应：**
```json
{
  "code": 200,
  "msg": "success",
  "data": null
}
```

**副作用：**
- `production_order.device_id / device_no / device_name` 更新为所选打印机
- `production_process_record`（process_type=PRINT）的 `device_id / device_no / device_name` 同步更新

### 4.8 提交生产批号

**POST** `/production/order/{id}/submit-batch-no`

**说明：** 用户填写（或自动生成后确认）生产批号后提交，同时可填写原材料批号。

**请求体：**
```json
{
  "productionNo": "20260513000001-1",
  "materialBatchNo": "MAT-2026-001"
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| productionNo | String | 是 | 生产批号 |
| materialBatchNo | String | 否 | 原材料批号 |

**响应：**
```json
{
  "code": 200,
  "msg": "success",
  "data": null
}
```

**副作用：**
- `production_order.production_no` 更新为提交值（需校验唯一性）
- `production_order.material_batch_no` 更新为提交值

#### production_product（生产产品表）

每条生产订单对应多个物理产品单元，在提交打印机配置（进入打印中）时由后端按 `design_product.quantity` 展开创建，每件产品一行。

```sql
CREATE TABLE production_product (
  id                    BIGINT        NOT NULL AUTO_INCREMENT COMMENT '主键ID',
  production_order_id   BIGINT        NOT NULL COMMENT '生产订单ID',
  production_no         VARCHAR(80)   NOT NULL COMMENT '生产批号（冗余）',
  design_product_id     BIGINT        NOT NULL COMMENT '来源设计产品ID（design_product.id）',
  product_name          VARCHAR(128)  COMMENT '产品名称（冗余）',
  spec_name             VARCHAR(128)  COMMENT '规格名称（冗余）',
  cert_no               VARCHAR(64)   COMMENT '注册证号（冗余）',
  material_name         VARCHAR(64)   COMMENT '材质名称（冗余）',

  -- ==================== 生产编号 ====================
  -- 格式：生产批号 + 序号，如 20260513000001-1-001
  product_code          VARCHAR(100)  COMMENT '生产编号（每件产品唯一）',

  -- ==================== UDI ====================
  -- 包装工序完成时生成，格式遵循 GB/T 42062
  udi_code              VARCHAR(200)  COMMENT 'UDI码（包装工序生成）',
  udi_di                VARCHAR(100)  COMMENT 'UDI-DI（设备标识符，来自注册证）',
  udi_pi                VARCHAR(100)  COMMENT 'UDI-PI（生产标识符：批号+序列号）',

  -- ==================== 质检 ====================
  -- 0=待质检, 1=合格, 2=不合格
  qc_result             TINYINT       NOT NULL DEFAULT 0 COMMENT '最终质检结果',

  sort_order            INT           DEFAULT 0 COMMENT '同订单内排序',

  create_time           DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP,
  update_time           DATETIME      NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
  create_by             BIGINT,
  update_by             BIGINT,
  is_deleted            TINYINT       NOT NULL DEFAULT 0,

  PRIMARY KEY (id),
```

### 5.2 完整状态流转

#### production_order.status

```
3010 设计完成
  └─[下载数据包(4.3)]→ 3020 待打印
      └─[提交打印机配置(4.6)]→ 3030 打印中
          └─[WebSocket PRINT_DONE，自动推进，无需人工]→ 3040 打印完成
              └─[WASH开始]→ 3050 后处理中
                  └─[WASH→CURING→ULTRASONIC 全部质检合格]→ 3060 后处理完成
                      └─[提交质检管理(POST /qc/order/submit-from-production)]→ 3065 终检中
                          └─[质检管理全部合格]→ 3066 终检完成
                              └─[PACK完成]→ 3070 入库

不合格分支：
  REDO_CURRENT     → order.status 不变，当前工序重置为 0
  REWORK_TO_PRINT  → order.status → 3030，PRINT 及之后所有工序重置为 0
```

> **PRINT 工序说明：** 提交打印机配置后，打印机自动开始打印。前端通过 WebSocket 监听打印进度，收到 `PRINT_DONE` 事件后后端自动将 PRINT 工序 status→4（待质检），前端展示质检入口。整个打印过程无需人工点击「开始」/「完成」。

#### production_process_record.status

```
PRINT 工序（WebSocket 驱动）：
  0 待执行 →[提交打印机配置，自动]→ 1 执行中 →[WebSocket PRINT_DONE，自动]→ 4 待质检
    →[质检合格]→ 2 已完成
    →[质检不合格 REDO_CURRENT]→ 0 待执行（重置，清空 process_config）
    →[质检不合格 REWORK_TO_PRINT]→ 0 待执行（本工序及之前所有工序重置）

后处理工序（WASH / CURING / ULTRASONIC / PACK，人工操作）：
  0 待执行 →[人工点击开始(4.11)]→ 1 执行中 →[人工点击完成(4.12)]→ 4 待质检
    →[质检合格]→ 2 已完成
    →[质检不合格 REDO_CURRENT]→ 0 待执行（重置，清空 process_config）
    →[质检不合格 REWORK_TO_PRINT]→ 0 待执行（本工序及之前所有工序重置）
```

#### 4.15 提交质检

**POST** `/production/order/{id}/process/qc`

**说明：** 质检以**产品为单位**逐个提交。同一工序下所有产品全部合格后，工序才推进到下一步。

**请求体（单个产品合格）：**
```json
{
  "processType": "WASH",
  "productId": 1,
  "qcResult": 1
}
```

**请求体（单个产品不合格）：**
```json
{
  "processType": "WASH",
  "productId": 2,
  "qcResult": 0,
  "failReason": "清洗不彻底，表面残留支撑材料",
  "handleType": "REDO_CURRENT"
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| processType | String | 是 | 工序类型 |
| productId | Long | 是 | 产品ID（design_product.id） |
| qcResult | Integer | 是 | 1=合格, 0=不合格 |
| failReason | String | 条件必填 | 不合格原因（qcResult=0时必填） |
| handleType | String | 条件必填 | REDO_CURRENT / REWORK_TO_PRINT（qcResult=0时必填） |

**handleType 枚举：**

| 值 | 说明 |
|----|------|
| REDO_CURRENT | 该产品重做当前工序 |
| REWORK_TO_PRINT | 该产品返工到打印 |

**响应：**
```json
{
  "qcRecordId": 201,
  "allProductsPassed": false,
  "pendingProductIds": [2],
  "nextAction": null,
  "nextProcessType": null
}
```

| 字段 | 说明 |
|------|------|
| allProductsPassed | 当前工序所有产品是否全部合格 |
| pendingProductIds | 尚未质检或不合格的产品ID列表 |
| nextAction | 全部合格时返回：NEXT_PROCESS；否则为 null |
| nextProcessType | 下一工序类型（全部合格且有下一工序时返回） |

**业务规则：**

| 场景 | 变更 |
|------|------|
| 单产品合格，仍有其他产品未合格 | 记录该产品质检结果，工序状态不变（继续待质检） |
| 所有产品全部合格，有下一工序 | `process_record.status` → 2，返回 nextAction=NEXT_PROCESS |
| 所有产品全部合格，ULTRASONIC 最后一道 | `process_record.status` → 2，`order.status` → 3060 |
| 单产品不合格 + REDO_CURRENT | 仅该产品重新执行当前工序（其他产品不受影响），工序整体回到执行中 |
| 单产品不合格 + REWORK_TO_PRINT | 仅该产品从 PRINT 工序重新开始，其他产品不受影响 |

**前置校验：** 当前工序 status=4（待质检）；qcResult=0 时 failReason 和 handleType 必填
